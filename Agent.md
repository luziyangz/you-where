# Agent 协作总纲

## 1. 目标
- 交付一个稳定、可靠、可维护、支持多并发、抗干扰的微信小程序系统。
- 范围覆盖原生微信小程序前端、Python FastAPI 后端、产品治理、测试验证、交付准备。

## 2. 角色编组
- 产品角色：负责需求澄清、页面闭环、验收标准、体验一致性、改进建议。
- 开发角色：负责原生微信小程序前端、Python 后端、接口契约、性能、稳定性、工程治理。
- 测试角色：负责测试策略、回归清单、冒烟、接口验证、异常链路与风险拦截。

## 3. 共享 Memory
- 共享记忆主文件：`team/shared-memory.md`
- 对话记录：`record.md`
- 待办事项：`待办.md`
- 已办事项：`已办.md`
- 错误记录：`team/error-log.md`
- 当前计划：`team/current-plan.md`

所有角色必须把“可复用的项目事实、决策、约束、接口认知、风险和结论”沉淀到共享文件中，而不是只停留在一次性对话里。

## 4. 工作节奏
1. 先审查，再计划。
2. 计划必须明确：目标、边界、可行性、风险、验证方式、完成标准。
3. 产品、开发、测试三方需要在每轮开始前对目标一致，在每轮结束后对结果一致。
4. 每轮结束必须更新 `record.md`、`待办.md`、`已办.md`、`team/current-plan.md`。

## 5. Hooks 机制
- Pre-Scan Hook：开始任何任务前，先确认范围、影响面、依赖、是否涉及前后端联动。
- Pre-Change Hook：改代码前检查引用关系、主流程、回归影响、是否有测试支撑。
- Post-Change Hook：改动后执行最小必要验证，至少覆盖主链路与直接影响面。
- Plan-Sync Hook：每轮计划结束时，同步产品、开发、测试视角的结论与差异。
- Error-Hook：同一问题累计出现 3 次及以上，必须登记到 `team/error-log.md`，写明现象、触发条件、影响、临时措施、根因与后续动作。

详见 `team/hooks.md`。

## 6. 升级与暴露原则
- 遇到无法回避的问题，直接暴露，不隐藏、不模糊描述。
- 无法立即解决时，必须给出：
  - 现象
  - 影响范围
  - 已尝试手段
  - 当前阻塞点
  - 下一步解决方案

## 7. 清理原则
- 仅清理“明确无引用、未接入主流程、可被历史记录替代”的目录与文件。
- 删除前先核对引用关系。
- 对可能仍有价值但不应留在主线的历史内容，优先留在 `archive`，不混入运行主链路。

## 8. Definition of Done
- 产品：功能目标、异常场景、用户体验和验收标准明确。
- 开发：实现完整，可维护，有最小必要验证，主链路不退化。
- 测试：核心链路冒烟通过，风险项已记录，未覆盖项明确。
- 文档：共享 memory、待办、已办、错误记录已同步。

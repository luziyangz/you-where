## 0. 产品背景与目标

### 0.1 背景

- 碎片时间阅读越来越多，但**「一起读同一本书」**的体验很弱：
  - 微信读书 / Kindle 等偏「个人阅读」，缺少简单直接的「两个人同步进度」工具。
  - 即使在用同一平台，双方也很难直观知道「你读到哪一页了」。
- 很多情侣 / 朋友会一起读书、一起看同一本漫画或 PDF，但会遇到：
  - 一方读得快，一方读得慢，容易剧透、失去同步感。
  - 临时想记录想法，只能发微信 / 备忘录，无法和进度绑定沉淀下来。

### 0.2 产品目标（MVP）

- **核心目标：** 提供一个「双人共读进度同步 + 防剧透 + 轻量互动」的小工具。
- **本期（v1.0.0 / MVP）要达成：**
  - 支持两个人通过共读码绑定关系。
  - 支持对「一本正在共读的书」进行：进度记录、笔记记录、同步展示。
  - 支持基础防剧透逻辑（超过对方进度的笔记自动锁定）。
- **不做的事情（MVP 外）：**
  - 不做书籍内容展示 / 电子书阅读器，只记录「页码」和「想法」。
  - 不做广场 / 社区 / 推荐，仅支持「二人世界」。
  - 不做复杂多端登录，只考虑微信小程序 + 后端存储。

### 0.3 目标用户

- 情侣 / 夫妻：一起读同一本书、漫画、小说，希望「同步进度 + 互动」。
- 好友 / 同学：一起读非虚构类书籍（学习类、专业类），希望有一点点互相督促。
- 轻量阅读者：不想学习复杂 app，只要一个「输入页码 + 看对方到哪」的小工具。

### 0.4 典型使用场景（User Story）

1. **异地情侣共读小说**
   - 小 A 和小 B 约好一起读同一本小说，各自在纸质书上看。
   - 每天看完一段，小 A 打开小程序，输入「今天读到 128 页」，顺便写一句笔记。
   - 小 B 打开时，能看到双方进度条和自己的落后 / 领先情况，部分笔记被锁，提示「读到第 200 页解锁」。

2. **备考 / 学习搭子**
   - 两个同学一起读《算法导论》，用总页数映射进度。
   - 每个人记录自己看完的页码，重要的知识点写在笔记里。
   - 没看完的那一位，看不到超出自己当前进度的笔记，避免提前剧透理解路径。

> 说明：以上场景主要是为了让团队在讨论需求 / 迭代时，有具体画面感。

## 1.x 版本规划与范围

### 1.x.1 当前版本范围（v1.0.0 / MVP）

- **包含：**
  - 单一共读关系（一个账号同一时间只绑定一名伙伴）。
  - 单书共读（一个共读关系下同一时间只共读一本书）。
  - 页码级进度记录 + 笔记 + 防剧透锁定。
  - 基础成就统计（书本数量、页数、笔记数）。
  - 微信登录、基本隐私与数据安全。
- **不包含：**
  - 多人共读（N > 2）。
  - 书籍封面 / ISBN 自动抓取、书城 / 搜索。
  - 推送消息（模板消息 / 订阅消息），目前仅在小程序内用红点 + 震动提示。
  - 数据导出 / 分享海报。

### 1.x.2 成功标准（针对开发 / 验证）

- 功能上：两名用户可以在一天内完成以下链路：
  1. 创建账号 -> 绑定为共读伙伴。
  2. 添加一本书为「正在读」。
  3. 双方各自至少记录 2 次进度 + 笔记。
  4. 被防剧透锁定的笔记，在进度追平后自动解锁。
  5. 双方都勾选「我已读完」，书籍进入「已读完」列表。
- 体验上：
  - 常见异常输入（页码错误、断网）有明显提示、不丢数据。
  - 登录状态在正常使用中不会频繁失效。

# 《你在哪页》共读小程序产品需求文档 (PRD)

**版本：** v1.0.0 (MVP)  
**目标：** 实现双人同步阅读进度，通过“进度盲盒”机制提升共读动力。

## 1. 产品概述

“你在哪页”是一款专注双人（情侣、好友）共读场景的同步工具。用户通过绑定唯一共读码，实现阅读进度可视化、想法实时互动，并利用“进度锁”机制防止剧透，增加阅读过程中的惊喜感。

## 2. 角色与权限

| 角色 | 权限说明 |
| :--- | :--- |
| **本人 (Me)** | 管理书籍、更新个人进度、发布笔记、回复伙伴、解绑伙伴。 |
| **共读伙伴 (Partner)** | 与本人通过代码绑定，其进度和笔记对本人可见（受逻辑锁限制）。 |

---

## 3. 功能详细说明

### 3.1 关系管理模块 (Relationship Management)

#### 3.1.1 身份标识与绑定

- **我的共读码：** 系统为每个用户分配唯一的 6 位数字 ID。
- **绑定逻辑：**
  - 输入对方 6 位码即可直接绑定。
  - **排他性：** 一个用户在同一时间内只能绑定一名共读伙伴。
  - **联动：** 绑定后，首页即时显示对方头像、昵称及当前共读进度。
- **解绑逻辑：**
  - 任何一方均可随时发起解绑。
  - 解绑后，过往所有共读进度数据保留在个人历史中，但不再更新对方数据，且“共读流”互动功能关闭。

#### 3.1.x 绑定前置条件与失败情况

- 绑定前置条件：
  - 双方均已完成登录。
  - 双方当前都「未绑定其他伙伴」。
- 失败情况（需有明确前端提示文案）：
  - 输入的共读码不存在：提示「未找到对应用户，请确认对方共读码是否正确」。
  - 对方已绑定他人：提示「对方已与其他伙伴共读，无法绑定」。
  - 自己输入了自己的共读码：提示「不能与自己绑定」。

#### 3.1.x 解绑后的界面表现

- 解绑成功后：
  - 首页「伙伴信息区」恢复为「未绑定状态」样式。
  - 「正在读」中的共读信息仅保留本人部分，对方数据不再更新。
  - 个人历史中保留既有共读记录，但不再标注为「共读中」，可标记为「曾与 XX 共读」（可选，MVP 可先不做）。

---

### 3.2 书籍管理模块 (Book Management)

#### 3.2.1 正在读 (Reading Tab)

- **数量限制：** 为保证共读专注度，一个共读关系下，**同一时间只能有一本“正在读”的书籍**。
- **添加书籍：**
  - 需录入：书名（必填）、作者（选填）、总页数（必填）。
  - 逻辑校验：若已有“正在读”的书籍，必须先将其标记为“已读完”，方可添加新书。
- **卡片信息展示：**
  - 双人进度对比条（我 vs 伙伴）。
  - 进度标签：实时计算“伙伴领先 / 落后 X 页”。
  - 状态标签：若伙伴未加入，显示“等待加入”。

#### 3.2.2 已读完 (Finished Tab)

- **归档逻辑：** 当且仅当**双方**都勾选了“我已读完”时，书籍自动进入“已读完”列表。
- **历史回溯：** 用户可以点击已读完的书籍，查看完整的共读流笔记记录。

---

### 3.3 共读流与进度更新 (Core Progress System)

#### 3.3.1 进度可视化仪表盘

- **进度映射：** 分别展示两人的实时进度页码。
- **游标互动：** 当两人页码重合或接近（距离 < 2%）时，游标可略微错位显示，避免完全遮挡。

#### 3.3.2 笔记发布与进度更新

- **触发动作：** 用户点击“记一笔 / 更新”。
- **逻辑校验：**
  - 输入的页码不能小于当前已记录的页码。
  - 填写的页码不能大于书籍总页数。
- **阅读完毕状态：** 用户可手动勾选“我已读完”，此时页码强制更新为总页数。

#### 3.3.3 防剧透锁定逻辑 (Spoiler Lock) —— **核心功能**

- **规则定义：**
  - 设本人当前进度为 \(P_{me}\)，伙伴发布的笔记所在页码为 \(P_{note}\)。
  - 若 \(P_{note} > P_{me}\)：笔记内容显示为“锁定”状态。
  - 若 \(P_{note} \le P_{me}\)：笔记内容自动解锁可见。
- **交互反馈：** 用户点击锁定笔记，提示“需读到 P(x) 页解锁”，并伴随震动反馈（Haptic Feedback）。

#### 3.3.4 回复互动

- **功能：** 用户可对已解锁的笔记进行即时回复。
- **提醒：** 对方更新进度或回复笔记时，“进度”模块有小红点提示，同时在进度页面，有定位图标，点击一键定位到更新进度或回复笔记位置，定位到时新消息同步震动一次；有多条消息时定位图标可多次定位；用户查看所有未读消息后定位图标和消息提示（小红点）消失。

#### 3.3.x 进度更新频率与幂等性

- 为降低误触、测试数据污染：
  - 允许用户频繁更新页码，但后端每条记录需有时间戳。
  - 同一用户在短时间内连续多次更新，仅作为多条记录存在，不做合并（方便以后做「回顾今天看了几次」）。
- 幂等性：
  - 若发生网络重试（前端重复提交同一条记录），后端可通过「客户端生成的 requestId」或「时间戳 + 用户 id + 页码」避免重复写入（可选，若实现成本高，MVP 阶段可接受极少量重复数据）。

#### 3.3.x 笔记内容限制

- 字数限制：建议 0～200 字，超过时前端直接限制输入。
- 文本内容仅支持纯文本（不支持图片 / 语音），避免复杂度。
- 敏感词过滤：MVP 阶段可不做全量敏感词库，但保留后端扩展能力。

---

### 3.4 个人中心与统计 (Profile)

- **成就数据：**
  - 累计读过：统计本人“已读完”列表的书籍总数。
  - 累计页数：统计所有书籍中本人贡献的页数总和。
  - 累计笔记：统计本人发布的笔记总数。
- **动画效果：** 数字增加时采用滚动动画，增强成就感。

---

### 3.5 登录与用户账号模块 (Authentication & User Identity)

#### 3.5.1 登录流程设计 (Authentication Flow)

- **静默登录：** 用户首次打开小程序时，调用 `wx.login` 获取 `code` 后端换取 `openID`，自动创建账号，实现“即开即用”。
- **授权登录：**
  - 当用户需要同步数据（如：发布笔记、绑定伙伴）时，引导用户进行“授权登录”。
  - **微信一键登录：** 采用微信标准头像昵称填写组件。
  - **数据合并：** 确保同一 `openID` 下的本地临时数据（如有）与云端账号数据实现无缝合并。

#### 3.5.2 用户资料管理 (Profile Management)

- **头像与昵称：**
  - 根据微信最新规范，支持用户自定义从微信头像选择或上传。
  - 默认昵称：设置为“书友_后四位 UID”。
- **共读码生成 (Join Code)：**
  - **生成规则：** 账号创建时，系统自动分配一个**唯一、永久**的 6 位数字码（如：829 104）。
  - **唯一性：** 全局唯一，不随退出登录或更换设备改变。
  - **展示：** 在“首页 - 伙伴管理区”和“我的”页面显著位置展示。
  - **交互：** 点击共读码自动复制到剪贴板，方便发送给微信好友。

#### 3.5.3 登录状态与持久化 (Session & Persistence)

- **Token 机制：** 采用有效期为 30 天的 Token 机制。过期前自动刷新，确保用户在使用过程中不会突然被迫重新登录。
- **设备关联：** 同一个账号支持在不同手机登录，登录后自动同步当前“正在读”的书籍进度和共读流记录。

#### 3.5.4 退出登录逻辑 (Logout Logic)

- **动作定义：** 用户在“我的”页面点击“退出登录”。
- **数据保护：** 退出登录仅清除本地缓存的 Token 和当前用户信息，**不解绑**伙伴关系，**不删除**云端阅读记录。
- **退出后状态：** 页面返回至“我的”页初始态（显示“点击登录”），隐藏所有统计数据。

#### 3.5.5 特殊场景处理 (Edge Cases)

- **未登录访问：** 允许未登录用户查看“首页”和“我的”页框架，但点击任何涉及数据的操作（如“+”号添加书籍、点击进度页、复制共读码）均需触发登录引导。
- **更换微信：** 若用户更换微信账号（新的 openID），系统将其视为新用户。若需找回旧账号，需提供原共读码并进行人工 / 特定验证（根据后续运营需求决定是否支持）。

---

### 3.6 隐私与合规 (Privacy & Compliance)

- **隐私协议确认：** 首次登录前，必须弹出《用户协议》与《隐私保护指引》勾选框。
- **数据脱敏：**
  - 在未绑定前，任何用户无法通过共读码反查对方的手机号等敏感信息。
  - 共读流中的笔记仅对绑定的双方可见。

---

### 交互原型补充说明 (UI/UX Supplementary)

1. **登录前态（首页）：**
   - “我的共读码”区域显示为“登录后可见”。
   - 绑定按钮点击后弹出“请先登录”提示框。
2. **登录中态：**
   - 调用微信原生授权半屏弹窗。
3. **登录后态（首页）：**
   - 实时拉取云端数据，若已有绑定伙伴，直接显示伙伴头像及当前共读进度。
4. **“我的”页动态效果：**
   - 登录成功后，累计书籍、页数、笔记数应从 0 开始跳动至实际数值（增加激励反馈）。

## 4. 非功能性需求与交互规范

### 4.1 异常流程处理

- **页码冲突：** 若用户输入的页码超过总页数，输入框需红框抖动提示并限制提交。
- **网络异常：** 发布笔记时若失败，需在本地保留草稿或提示重试，防止内容丢失。

### 4.1.x 空状态说明（更具体）

- 未绑定伙伴：
  - 首页显示插画 + 引导文案：「发送你的共读码给 TA，一起开始共读吧」。
  - 显示「复制共读码」按钮和「输入对方共读码」入口。
- 已绑定但未添加书：
  - 在进度区域显示「还没有正在共读的书」+「+ 添加一本」按钮。
- 已绑定且有正在读的书，但双方都暂未写任何笔记：
  - 进度条正常显示，但「共读流」区域显示「还没有任何想法，可以从这里记一笔」。

### 4.1.x 网络重试与草稿

- 发布笔记失败时：
  - 前端需保留当前输入内容（不清空输入框）。
  - 显示「网络异常，稍后重试」提示，并允许用户再次点击提交。
- 若用户离开页面：
  - 可选：MVP 阶段可以不做「自动草稿恢复」，但需要评估实现成本。如果实现不复杂，可以将最近一次未成功发送的内容暂存本地，下次打开时提示「发现未发送的笔记，是否继续编辑？」。

### 4.2 视觉与体验 (UI/UX)

- **配色方案：**
  - 本人：微信绿 (#07c160)
  - 伙伴：温暖橙 (#ff9f00)
- **空状态：** 当“正在读”为空时，首页展示引导插画及“+”按钮。
- **隐私：** 只有绑定的共读伙伴能看到笔记，不支持任何形式的公开分享至广场。

### 4.3 数据安全性

- **数据存储：** 使用微信小程序云数据库，实时同步双人数据。
- **权限校验：** 每次请求笔记流时，后端需校验请求者与笔记所有者的绑定关系。

## 5. 技术与数据（简要）

### 5.1 核心数据实体（抽象）

> 说明：本节为概念层数据结构说明，**实际 API JSON 字段统一使用 `snake_case`**，如 `user_id`、`join_code`、`total_pages`，详见第 6.8 节接口风格约定。

- 用户（User）
  - user_id（open_id 映射）
  - join_code（6 位共读码）
  - nickname / avatar
- 共读关系（Pair）
  - pair_id
  - user_a_id / user_b_id
  - status（active / unbound）
- 书籍（Book）
  - book_id
  - pair_id
  - title / author / total_pages
  - status（reading / finished）
- 进度与笔记（Entry）
  - entry_id
  - book_id
  - user_id
  - page
  - note_content
  - created_at

### 5.2 防剧透判断（后端建议也做一层校验）

- 接口返回共读流时，后端根据：
  - 当前请求者进度 `P_me`
  - 每条笔记 `P_note`
- 为每条记录标记：
  - `isLocked = P_note > P_me`
- 前端根据 `isLocked` 决定显示「锁定态」或真实文字，避免前端绕过。

## 6. 实现说明（供前后端 / 测试使用）

本节不追求完整 API 文档，而是给出关键实现思路与最小接口清单，方便前后端协同与测试设计。

### 6.1 小程序整体架构

- **前端技术栈：**
  - 微信小程序原生框架（WXML / WXSS / JS / JSON），按需使用微信云开发。
  - 使用全局 `App.globalData` 或自建轻量 store 管理：登录态（token）、用户信息、当前共读关系、当前书籍等。
- **页面与路由结构（对应 Tab）：**
  - `pages/home/index`：首页（伙伴状态 + 当前书籍概览）。
  - `pages/progress/index`：进度 / 共读流详情页。
  - `pages/profile/index`：我的 / 账号与成就页。
  - 其他弹窗 / 子页面：如「输入共读码绑定」「添加书籍」「记一笔」等，建议用组件 + 显隐控制实现，必要时独立页面。

### 6.2 登录与鉴权实现流程

- **静默登录流程（小程序启动时）：**
  1. 前端调用 `wx.login()` 拿到 `code`。
  2. 调用后端 `POST /api/v1/auth/login` 接口，传入 `{ code }`。
  3. 后端通过微信接口换取 `open_id`，查找或创建用户，生成 30 天有效期的 `token`。
  4. 返回数据：`{ token, user: { user_id, nickname, avatar, join_code, ... } }`。
  5. 前端将 `token` 与 `user` 缓存到本地（`wx.setStorage`）和全局状态，并更新 UI。
- **协议与隐私校验：**
  - 若后端发现用户尚未同意协议，可在返回中附带字段 `need_agreement = true`。
  - 前端据此弹出协议勾选弹窗，用户勾选后调用 `POST /api/v1/auth/accept-agreement` 更新 `agreement_accepted_at`。
- **Token 鉴权中间层（后端）：**
  - 对所有需要登录的接口，统一在中间件中解析请求头 `Authorization: Bearer <token>`。
  - 校验 token 后，将 `user_id` 注入到请求上下文，接口内部直接使用。

### 6.3 用户与共读关系接口示例

- **获取当前用户信息**
  - `GET /api/v1/me`
  - 返回：`{ user_id, nickname, avatar, join_code, ... }`
- **绑定伙伴**
  - `POST /api/v1/pair/bind`，请求体 `{ join_code }`（对方的 6 位码）。
  - 后端逻辑：
    - 校验当前用户与目标用户是否都未绑定他人。
    - 校验不能绑定自己。
    - 创建 / 更新 `Pair` 记录为 `active`，并返回 `{ pair_id, user_a, user_b }`。
  - 典型错误码：
    - `PAIR_NOT_FOUND`：共读码不存在。
    - `PAIR_ALREADY_BINDED`：对方已被绑定。
    - `PAIR_SELF_BIND`：不能与自己绑定。
- **解绑伙伴**
  - `POST /api/v1/pair/unbind`
  - 后端逻辑：将当前活动 `Pair` 标记为 `unbound`，不删除书籍与笔记数据。

### 6.4 书籍与进度相关接口示例

- **创建书籍**
  - `POST /api/v1/books`，请求体 `{ title, author, total_pages }`
  - 后端逻辑：
    - 查找当前用户所属的活动 `Pair`。
    - 校验该 `Pair` 是否已有状态为 `reading` 的书籍，若有则拒绝创建。
    - 创建 `Book` 记录并返回。
- **查询书籍列表**
  - `GET /api/v1/books?status=reading` 或 `status=finished`
  - 返回当前 `Pair` 下对应状态的书籍数组。
- **更新进度 / 记一笔**
  - `POST /api/v1/entries`，请求体 `{ book_id, page, note_content? }`
  - 后端逻辑：
    - 校验 `book_id` 属于当前 `Pair`，且状态为 `reading`。
    - 获取当前用户在该书上的最新进度 `current_page`，校验 `page >= current_page` 且 `page <= total_pages`。
    - 写入一条新的 `Entry` 记录（包含时间戳），返回最新进度。

### 6.5 防剧透与共读流实现

- **获取共读流**
  - `GET /api/v1/books/{book_id}/entries`
  - 后端步骤：
    1. 校验当前用户与 `book_id` 所属 `Pair` 是否匹配。
    2. 查出当前用户在该书上的最新进度 `P_me`。
    3. 查询该书下所有 `Entry`（可分页）。
    4. 对每条记录计算：`isLocked = entry.page > P_me`。
    5. 返回时，若 `isLocked = true`，可不返回 `noteContent`，仅返回 `maskedContent` 或由前端统一展示占位文案。
- **前端渲染**
  - 根据 `isLocked` 决定展示真实内容或锁定样式。
  - 点击锁定笔记时，本地弹出提示「读到第 X 页就能解锁 TA 的想法～」，并调用 `wx.vibrateShort()`。

### 6.6 回复与未读提醒实现

- **回复接口**
  - `POST /api/v1/entries/{entry_id}/replies`，请求体 `{ content }`
  - 后端将回复与 `Entry` 关联，记录 `userId` 与时间戳。
- **未读状态追踪**
  - 为每个用户和每本书维护一个「已读到的最后一条消息 ID」或时间戳 `last_read_at`。
  - 获取共读流时，对比每条 `Entry` / `Reply` 的时间戳，统计「未读数量」，在接口中返回 `unread_count`。
- **前端行为**
  - 若 `unread_count > 0`：在 Tab 上显示小红点，在进度页显示「定位」按钮。
  - 点击「定位」按钮时，滚动到下一条未读消息位置；全部阅读后清理本地未读标记，并调用后端更新 `last_read_at`。

### 6.7 成就统计实现

- **统计口径**
  - 累计读过书籍数：统计当前用户作为 `Pair` 成员，且书籍状态为 `finished` 的去重 `bookId` 数量。
  - 累计页数：统计所有 `Entry` 中，该用户在每本书的最大页码之和（每本书只取最大值）。
  - 累计笔记数：统计该用户创建的 `Entry` 条数。
- **接口**
  - `GET /api/v1/me/stats`
  - 返回：`{ total_books, total_pages, total_entries }`，供「我的」页面展示并做数值滚动动画。
